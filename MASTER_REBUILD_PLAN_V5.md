## 9. Broad Differentiator Doctrine

### 9.1 Global Rule

Differentiators are not limited to signal files.

A differentiator is **anything in POVERTY_KILLER that is intended to create unique real edge beyond ordinary baseline systems**, including:

- strategies
- alpha engines
- execution behavior
- market reaction logic
- market / instrument selection logic
- allocation logic
- sizing logic
- stop / attack / escalation behavior
- risk mitigation behavior
- evaluation and feedback loops
- orchestration/runtime edge
- structural data and state systems
- anything unique that materially improves decisions, routing, sizing, risk, execution, or survival

A differentiator must be:

- real
- operational
- behaviorally meaningful
- mathematically justified where applicable
- structurally integrated
- visibly useful to decisions, routing, sizing, risk, or execution

A differentiator that does not materially influence outcomes is not a true differentiator.

### 9.2 Real vs Decorative Classification Law

Every differentiator candidate must be classified as one of:

- REAL
- REAL BUT DISCONNECTED
- PARTIAL
- FAKE-INTEGRATED
- BROKEN
- STUB
- DEFERRED
- RETIRED

Definitions:

- **REAL** = behaviorally live and meaningfully consumed
- **REAL BUT DISCONNECTED** = real logic exists but is not wired into live behavior
- **PARTIAL** = meaningful logic exists but the component is incomplete or limited
- **FAKE-INTEGRATED** = appears wired, but wrong objects/default fallbacks/null paths mean it silently contributes nothing
- **BROKEN** = import/contract/runtime failure prevents lawful use
- **STUB** = placeholder or near-empty
- **DEFERRED** = intentionally postponed without current doctrine violation
- **RETIRED** = obsolete path, tombstone, or historical artifact

## 10. Split-Role Preservation Doctrine

When a differentiator contains two genuinely valuable and distinct forms of edge, do not force them into one compromised module.  
Split them into separate role-pure modules and preserve both.

Whale is the governing example.

However:

- do not split for elegance
- do not split merely because a file emits multiple outputs
- do not split an integrated mathematical engine if it is still one coherent model
- do not proliferate modules without real edge justification

### 10.1 Proof Burden for Non-Split Decisions

A possible second weapon may not be dismissed merely because it is hard to understand on first pass.

If a file appears to contain a secondary behavior, output stream, state machine, memory layer, contextual edge, or latent second weapon, the reviewer/coder must determine whether it is:

- genuinely useless
- merely decorative
- redundant with the primary role
- mathematically inseparable from the primary engine
- already lawfully represented inside the primary model without edge loss

If concluding that a split is **not** warranted, one of those justifications must be proved explicitly.

If that cannot be proved, removal, collapse, or simplification is forbidden.
