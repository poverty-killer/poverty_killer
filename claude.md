# 5. File Memory and Cost Discipline Law

You must use memory intelligently.

You must not waste time or money rereading files or regions that have already been read word-for-word and have not changed.

You must maintain a working in-session file memory ledger.

For every file inspected, record:

```text

```

## Read-once principle

If a file or exact source region was already read word-for-word in the current session and there is no evidence it changed, you must use that memory first.

Do not reread it just because a later bundle also references it.

Example:

If Task 1 required full reads of Module C and Module E, and later Task 3D also touches a downstream module that imports Module C and Module E, you must first use the existing file memory ledger for C and E.

You reread C or E only if one of these is true:

- the file changed since last read
- the Board uploaded a newer version
- command output shows changed hash / modified time / line count
- the prior read was partial and the missing region is now required
- the prior memory does not include the exact region needed
- the prior memory is ambiguous
- the patch target depends on exact current text not already captured
- an error proves the remembered contract may be stale
- the Board explicitly commands a fresh reread

## No permission needed for required reads

If a required file or required region has not been read sufficiently, you must automatically read it.

Do not ask permission to inspect required source truth.

Do not say: “I need to read X first, may I?”

Read it.

Then report what you read.

## No permission needed to expand read scope

If local context proves insufficient, automatically expand the read scope.

Do not ask permission to read additional required lines, imports, state definitions, callers, consumers, or contracts.

Read the smallest additional scope needed.

Then continue.

## Hash/modified-time discipline

When possible, check file size, modified time, or hash before rereading a previously read file.

If unchanged, rely on memory.

If changed, reread only the changed file or exact changed region first.

## Memory expiry

In-session memory is valid until:

- file changed
- context is reset
- Board uploads newer source
- patch failure suggests stale understanding
- exact text is required and not already captured
- session memory becomes uncertain

Across a new terminal session, memory is valid only if restored from a Board-approved repo memory ledger, repo map packet, source packet, or explicit continuity packet.

If no such packet exists, inspect the required files or regions automatically.

# 6. Source Truth Sufficiency Law

You do **not** always need to read the entire file before patching.

You must read the **minimum sufficient source truth** required to patch safely.

Minimum sufficient source truth means:

1. exact target region to be patched
2. surrounding local context above and below the target region
3. relevant imports used by the target region
4. relevant class/function signature containing the target region
5. relevant state fields touched by the target region
6. relevant direct callers or consumers if behavior, contract, output, state, timing, risk, execution, strategy, telemetry, or authority changes
7. relevant contracts/enums/models if the patch touches payload shape, field names, object construction, validation, or downstream consumption
8. relevant tests or telemetry surfaces if the patch changes observable behavior

## Full-file read is required only when:

- the patch changes file-level authority
- the patch changes class initialization/state ownership
- the patch changes imports used across the file
- the patch changes public contract outputs
- the patch changes execution authority
- the patch changes risk authority
- the patch changes strategy authority
- the patch changes fusion authority
- the patch changes truth-frame authority
- the patch changes sizing authority
- the patch may affect multiple methods
- the file has hidden state interactions not visible from the target region
- prior local patch attempts failed
- the target region cannot be understood without broader context
- the Board explicitly requires full-file review
- the file memory ledger has no reliable prior full read and the module is authority-bearing
- the file is a protected differentiator and the change alters semantics, state, confidence, veto, sizing, routing, or execution effect

## Partial/local read is allowed when:

- the patch is narrow and text-local
- the target region has exact anchors
- imports/signatures/state dependencies are known or checked locally
- no public contract changes occur
- no authority ownership changes occur
- no cross-method state behavior changes occur
- direct caller/consumer impact is absent or already known
- the patch can be verified by targeted tests/import checks
- the change is a local bug fix, typo fix, logging improvement, guard hardening, or narrow adapter repair that does not alter broad semantics

## Before reading a whole file, ask internally:

“Can this patch be made safely from exact target-region context plus known imports/state/callers/consumers?”

If yes:

- read only the target region and required adjacent truth
- patch surgically
- verify targeted invariant

If no:

- read the whole file or the missing sections automatically

## If using partial/local read, you must state:

```text
source_truth_scope: partial/local
why_full_file_read_not_required:
target_region_read:
context_lines_read:
imports_checked: yes/no/not-needed
state_checked: yes/no/not-needed
callers_checked: yes/no/not-needed
consumers_checked: yes/no/not-needed
contracts_checked: yes/no/not-needed
tests_checked: yes/no/not-needed
risk_of_hidden_interaction:
verification_needed:
```

If uncertainty remains, expand the read scope automatically.

Do not ask permission to expand read scope when required.

Do not patch from blind snippets.

PostToolUse hook: python .claude/hooks/post_tool_use.py
- Fires after every Edit, Write, MultiEdit tool call

## Override protocol

Set POVERTY_KILLER_OVERRIDE=true and POVERTY_KILLER_OVERRIDE_REASON=<reason>.

Override reason rules:
- Must be present and non-blank
- Must be at least 20 non-whitespace characters
- Must not start with "test" (case-insensitive)

Override does not unlock live mode, pip install, destructive git,
or destructive delete commands. Those are hard-blocked always.

Every override attempt (success or failure) is logged to state/override_log.jsonl.
